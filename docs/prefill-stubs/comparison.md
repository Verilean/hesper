Source of llama.cpp side: `docs/prefill-stubs/overview.md` (auto-generated by `lake exe stub-extract --report`).
Source of hesper side: `Hesper/Layers/Linear.lean` (production), with line-anchored references.

Goal: identify what hesper is missing or doing differently from llama.cpp at the prefill kernel level, prioritized by impact on the **9.2× pp70 gap** documented in `project_prefill_9x_gap_2026_04_29.md`.

### Gap identified

1. **MMQ5 is implemented but gated behind `HESPER_PREFILL_MMQ5=1` and not default**. MMQ2 is default.  Promoting MMQ5 to default is the single biggest win available — should close most of the 9.2×.
2. **`__launch_bounds__(..., 2)` (≥ 2 blocks/SM) hint is missing**. Without it, the compiler may pick a higher register budget that drops occupancy. Adding the equivalent in ShaderM (or a CodeGen hint) lets the kernel run with 2 blocks/SM as on llama.
3. **No stream-K variant**. llama.cpp uses stream-K (`grid=(nsm,1,1)`) when `mmq_x` is small relative to total tile work, splitting K across SMs and issuing a separate `stream_k_fixup` kernel. hesper does not. Lower priority — only matters when `mmq_y * mmq_x` per-WG is small.
4. **Inner dot path: dp4a vs int8 mma.sync (Tensor Core)**. llama selects mma.sync on Turing+. hesper only has dp4a. Blocked by `feedback_nvcc_ptx_version_vs_driver.md` (PTX 8.7 vs driver 565).

## 2026-05-02 PTX diff result

`HESPER_PTX_DUMP=/tmp/hesper_ptx` で hesper MMQ5 PTX 抽出, llama.cpp PTX は `/tmp/llamacpp_ptx/mmq_q4k.ptx` の `_Z9mul_mat_qIL9ggml_type12ELi64ELb0EE` entry を切り出して比較:

| metric | hesper MMQ5 | llama mmq_x=64 | 比 |
|---|---:|---:|---:|
| PTX 行数 | **16,473** | 640 | **25.7×** |
| PTX bytes | 525 KB | 21 KB | **25×** |
| `.reg .b32` | 14,873 | 3,555 | 4.2× |
| `.reg .pred` | 172 | 66 | 2.6× |
| `dp4a` 命令 | **256** | 0 | — |
| `ld.shared` | **960** | 0 | — |
| `$L_*` ループ | **0** | 19 | — |

### 解釈

**hesper MMQ5 は static unroll しすぎて巨大コードを生成している**。`for jIter in [0:8]` と `for iIter in [0:4]` (Lean static) が PTX レベルで 32 倍のコード展開を引き起こす。256 個の dp4a + 960 個の ld.shared が **すべてベタ書きで PTX に並んでいる** (内部 ShaderM.loop による圧縮が効いていない)。

llama.cpp は対照的に **runtime ループ + inline asm 経由の dp4a で 600 行に収まる**. 同じ計算量を実現するために kernel 体積は **25 倍小さい**. これは:

- PTX JIT cost が 25× 小さい (hesper の 3.5s cold が llama では ~150ms 級になる)
- ICache 圧迫が小さい → instruction fetch latency 軽減
- ptxas optimization が通りやすい (大きすぎると CSE/coalesc fail する)

### Action: hesper の static unroll を runtime loop に書き換える

Lean の `for jIter in [0:8] do ...` と `for iIter in [0:4] do ...` を **`ShaderM.loop` (runtime CUDA loop) で書き直す** 必要がある。ただし accumulator が `acc_{jIter}_{iIter}` の static name 32 個に依存しているので、accumulator 配列化 (smem array または register-based indirection) も同時に必要。

工数: ~250 行 / 2-3 セッション。

### 2026-05-02 Stage H stub-driven 確認

stub extractor に `#pragma unroll` 検出を追加 (`[Y-coop, unroll]` / `[K-outer]` (no tag = runtime) のように loop タグ拡張). llama.cpp の loop unroll 戦略が読める:

| llama.cpp loop | tag |
|---|---|
| `mul_mat_q`'s `for (j0...) ` | `[Y-coop, unroll]` ← coop load なので展開 OK |
| `mul_mat_q_process_tile`'s `for (kb0 = kb0_start; kb0 < kb0_stop; ...)` | `[K-outer]` (タグ無し = **runtime**) ← **PTX 600 行の理由** |
| `mul_mat_q_process_tile`'s `for (l0...)` | `[tile-coop, unroll]` ← coop load なので展開 |
| `flash_attn_ext_f16_process_tile`'s `for (; kb0 < kb0_stop-1; ++kb0)` | `[K-outer]` ← runtime |

ここから読める設計原則: **K-outer は runtime, coop loads/inner accumulator は unroll**.

hesper MMQ5 では:
- K-outer (`kbx0`) は ShaderM.loop (= runtime) ← ✓ llama と同じ
- pairIdx, elemOff も ShaderM.loop ← ✓
- **`for jIter in [0:8]` と `for iIter in [0:4]` は Lean static unroll ← ✗** これが 32× コード膨張の主因

llama.cpp の対応する箇所は inline asm 経由 mma 命令で「runtime loop で trip count に比例した命令数」になっている。

### 中間ステップ案: MMQ3 派生

MMQ5 をフルに直す (250 行) 前に、tile を半分にした **MMQ3** (mmq_y=64, mmq_x=32) を作る:
- accumulator `2 × 4 = 8 個 / thread` (32 → 8) → register 完全収納
- Lean static unroll は `jIter=4, iIter=2` で 8 倍 (現 MMQ5 32 倍 → 8 倍)
- PTX サイズが 1/4 程度に圧縮される見込み
- effort 100 行

これで「shape 縮めると速度どうなる」を実証できる。MMQ5 完全 fix の中間検証として。

### 代替

**既存 MMQ5 を捨てて llama.cpp PTX をそのまま raw load** する path を作る (#342 で wired 済). PTX 8.7 vs driver 565 制約は残るが、driver 更新で解ける。これが本筋とも言える。
